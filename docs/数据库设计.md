# 织梦纪 DreamWeave 数据库设计

版本：v0.2  
更新日期：2026-05-30

## 1. 设计目标

第一版数据库只服务最小互动闭环：

```text
保存用户
保存小说项目
保存会话
保存用户输入
保存 AI 回复
保存 AI 任务状态
```

Agent MVP 阶段需要在最小互动闭环基础上补充：

```text
保存 StoryState
保存长期剧情摘要
保存角色、关系、世界事件、物品等状态
保存 Worker 返回的状态更新结果
```

暂不设计复杂长期记忆向量库、推荐系统、支付、多人在线和复杂多 Agent 状态。

---

## 2. 技术选择

第一版使用：

```text
PostgreSQL
```

原因：

- 关系结构清晰，适合用户、故事、会话、消息等核心数据。
- `jsonb` 方便保存 AI 上下文、生成参数、任务元数据和 StoryState。
- 后续可以平滑接入全文搜索、向量扩展或任务审计。

---

## 3. ID 策略

第一版使用应用层生成的 `text` ID。

示例：

```text
user_local
story_local
session_local
msg_xxx
task_xxx
```

原因：

- 当前 server / Worker 已经使用字符串 ID。
- 本地开发、调试和日志排查更直观。
- 后续如果需要 UUID，可以单独做迁移。

---

## 4. 核心实体关系

```text
users 1 ── n stories
users 1 ── n sessions
stories 1 ── n sessions
sessions 1 ── n messages
sessions 1 ── n ai_tasks
sessions 1 ── 1 story_states
ai_tasks 1 ── 0..n messages
```

解释：

- 一个用户可以创建多个小说项目。
- 一个小说项目可以有多个会话。
- 一个会话包含多条消息。
- 每次 AI 生成会创建一条 AI 任务记录。
- 一个会话对应一份当前 StoryState。
- AI 任务成功后通常会生成一条 `assistant` 消息。

## 5. 表结构

### 5.1 users

用户表。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `text` | 用户 ID，主键 |
| `email` | `text` | 邮箱，可为空，唯一 |
| `nickname` | `text` | 昵称 |
| `created_at` | `timestamptz` | 创建时间 |
| `updated_at` | `timestamptz` | 更新时间 |

第一版本地开发可以固定使用：

```text
user_local
```

### 5.2 stories

小说项目表。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `text` | 小说项目 ID，主键 |
| `user_id` | `text` | 所属用户 |
| `title` | `text` | 小说标题 |
| `world_setting` | `text` | 世界观设定 |
| `character_setting` | `text` | 主角或用户角色设定 |
| `default_model` | `text` | 默认模型 |
| `generation_options` | `jsonb` | 默认生成参数 |
| `description` | `text` | 平台化作品简介，可为空 |
| `cover_image` | `text` | 作品封面地址，可为空 |
| `tags` | `text[]` | 作品标签 |
| `status` | `text` | 作品状态：`draft` / `published` / `archived` |
| `opening_message` | `text` | 玩家开始游玩时写入的开场白，可为空 |
| `author_id` | `text` | 作者 ID，第一版可等同 `user_id` |
| `created_at` | `timestamptz` | 创建时间 |
| `updated_at` | `timestamptz` | 更新时间 |

平台化 MVP 阶段暂时复用 `stories` 作为 `Work`，不立刻新建独立 `works` 表。玩家大厅读取 `status = 'published'` 的 stories，`POST /api/works/:id/play` 会基于 story 创建独立 session。

### 5.3 sessions

会话表。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `text` | 会话 ID，主键 |
| `user_id` | `text` | 所属用户 |
| `story_id` | `text` | 所属小说项目 |
| `title` | `text` | 会话标题 |
| `status` | `text` | 会话状态 |
| `created_at` | `timestamptz` | 创建时间 |
| `updated_at` | `timestamptz` | 更新时间 |

`status` 可选值：

```text
active
archived
deleted
```

### 5.4 messages

消息表。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `text` | 消息 ID，主键 |
| `session_id` | `text` | 所属会话 |
| `role` | `text` | 消息角色 |
| `content` | `text` | 消息内容 |
| `model` | `text` | AI 回复使用的模型，用户消息可为空 |
| `task_id` | `text` | 关联 AI 任务，可为空 |
| `metadata` | `jsonb` | 扩展信息 |
| `created_at` | `timestamptz` | 创建时间 |

`role` 可选值：

```text
user
assistant
system
```

第一版读取历史记录时，按：

```sql
session_id + created_at + id
```

排序。

### 5.5 ai_tasks

AI 任务表。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `text` | 任务 ID，主键 |
| `task_type` | `text` | 任务类型 |
| `status` | `text` | 任务状态 |
| `user_id` | `text` | 用户 ID |
| `story_id` | `text` | 小说项目 ID |
| `session_id` | `text` | 会话 ID |
| `worker_id` | `text` | 执行任务的 Worker |
| `request_id` | `text` | 协议请求 ID |
| `model` | `text` | 实际使用模型 |
| `input` | `text` | 用户输入 |
| `context` | `jsonb` | 下发给 Worker 的上下文 |
| `generation_options` | `jsonb` | 生成参数 |
| `output` | `text` | AI 输出内容 |
| `error_code` | `text` | 错误码 |
| `error_message` | `text` | 错误信息 |
| `retryable` | `boolean` | 是否可重试 |
| `timeout_ms` | `integer` | 任务超时时间 |
| `duration_ms` | `integer` | 执行耗时 |
| `created_at` | `timestamptz` | 创建时间 |
| `started_at` | `timestamptz` | 开始时间 |
| `completed_at` | `timestamptz` | 完成时间 |
| `updated_at` | `timestamptz` | 更新时间 |

`task_type` 第一版只要求：

```text
story_continue
```

`status` 可选值：

```text
created
sent_to_worker
running
success
error
timeout
cancelled
```

### 5.6 story_states

故事状态表。

用于保存互动小说 Agent 的长期状态。该表不是消息历史的替代品，而是当前故事世界的“结构化存档”。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `session_id` | `text` | 会话 ID，主键 |
| `user_id` | `text` | 所属用户 |
| `story_id` | `text` | 所属小说项目 |
| `state` | `jsonb` | 当前故事状态 |
| `version` | `integer` | 状态版本 |
| `created_at` | `timestamptz` | 创建时间 |
| `updated_at` | `timestamptz` | 更新时间 |

建议第一版 `state` 结构：

```json
{
  "current_world": "default_world",
  "current_scene": "",
  "story_stage": "opening",
  "long_summary": "",
  "characters": {},
  "relationships": {},
  "world_flags": {},
  "inventory": [],
  "pending_events": [],
  "user_preferences": {},
  "style_profile": {}
}
```

不建议长期保存到 `state` 的字段：

- `user_input`
- `recent_history`
- `retrieved_lore`
- `final_response`

这些字段属于单次 Agent 运行状态，应从请求、`messages`、Lore 检索结果或 `ai_tasks` 中获得。

### 5.7 lore_entries

世界观条目表。平台化 P4 阶段先绑定到 `stories.id`，后续如果拆分 `works` / `worlds`，再迁移到 `work_id` 或 `world_id`。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `text` | Lore 条目 ID，主键 |
| `story_id` | `text` | 所属作品 / story |
| `category` | `text` | 分类：`lore` / `character` / `location` / `faction` / `rule` / `style` / `event` |
| `title` | `text` | 条目标题 |
| `keywords` | `text[]` | 关键词 |
| `content` | `text` | 条目正文 |
| `priority` | `integer` | 优先级，第一版建议 0-100 |
| `enabled` | `boolean` | 是否启用 |
| `metadata` | `jsonb` | 扩展信息 |
| `created_at` | `timestamptz` | 创建时间 |
| `updated_at` | `timestamptz` | 更新时间 |

建议新增 migration：

```sql
CREATE TABLE IF NOT EXISTS story_states (
  session_id text PRIMARY KEY REFERENCES sessions(id) ON DELETE CASCADE,
  user_id text NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  story_id text NOT NULL REFERENCES stories(id) ON DELETE CASCADE,
  state jsonb NOT NULL DEFAULT '{}'::jsonb,
  version integer NOT NULL DEFAULT 1,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS idx_story_states_story
  ON story_states(story_id, updated_at DESC);
```

## 6. 第一版写入流程

### 6.1 用户发送消息

```text
1. 确保 user / story / session 存在
2. 插入 user message
3. 创建 ai_task，状态为 created
4. Server 发送任务给 Worker，状态更新为 sent_to_worker
5. Worker 返回结果
6. 更新 ai_task 为 success 或 error
7. 如果成功，插入 assistant message
8. 更新 session.updated_at
```

### 6.2 Agent 化后的用户发送消息

```text
1. 确保 user / story / session 存在
2. 读取 session 最近 10-20 条 messages
3. 读取或初始化 story_states.state
4. 插入 user message
5. 创建 ai_task，状态为 created
6. Server 组装包含 recent_messages 和 story_state 的 ai.task
7. Server 发送任务给 Worker，状态更新为 sent_to_worker
8. Worker 返回 content + state_update
9. 更新 ai_task 为 success 或 error
10. 如果成功，插入 assistant message
11. 合并 state_update 到 story_states.state
12. 更新 session.updated_at
```

### 6.3 页面加载历史

```text
1. 查询当前用户的 stories
2. 查询 story 下的 sessions
3. 查询 session 下的 messages
4. 按 created_at 升序显示
```

### 6.4 StoryState 合并原则

第一版可采用浅合并策略：

```text
旧 state
  ↓
合并 Worker 返回的 state_update
  ↓
写回 story_states.state
```

需要注意：

- `state_update` 只能覆盖允许更新的字段。
- `world_flags`、`characters`、`relationships` 等对象字段应按 key 合并。
- `pending_events`、`inventory` 等数组字段第一版可直接覆盖，后续再做追加 / 删除语义。
- 每次保存后 `version + 1`。

## 7. 第一版 API 对数据库的影响

当前接口：

```text
POST /api/story/continue
```

接入数据库后需要：

- 请求开始时保存用户消息。
- 创建 AI 任务记录。
- AI 成功后保存 assistant 消息。
- AI 失败后保存任务错误。
- 返回给前端时附带 `session_id`、`story_id`、`task_id`。

Agent 化后还需要：

- 请求前读取最近消息并注入 Worker context。
- 请求前读取或初始化 `story_states.state`。
- AI 成功后保存 Worker 返回的 `state_update`。
- 返回给前端时可附带 `state_update` 或当前 `story_state` 摘要。

后续需要新增：

```text
GET /api/stories
GET /api/stories/:story_id/sessions
GET /api/sessions/:session_id/messages
POST /api/stories
POST /api/sessions
GET /api/story/state/:session_id
PUT /api/story/state/:session_id
```

平台化 MVP 已新增：

```text
GET /api/works
GET /api/works/:id
POST /api/works/:id/play
```

作者后台 P3 / P4 已新增：

```text
GET    /api/creator/works
POST   /api/creator/works
GET    /api/creator/works/:id
PUT    /api/creator/works/:id
DELETE /api/creator/works/:id
POST   /api/creator/works/:id/publish
POST   /api/creator/works/:id/unpublish

GET    /api/creator/works/:story_id/lore
POST   /api/creator/works/:story_id/lore
PUT    /api/creator/works/:story_id/lore/:id
DELETE /api/creator/works/:story_id/lore/:id
```

第一版映射：

```text
Work        → stories where status = 'published'
PlaySession → sessions
StoryState  → story_states
```

## 8. 索引策略

第一版需要的索引：

- `stories(user_id, created_at)`
- `sessions(user_id, updated_at)`
- `sessions(story_id, updated_at)`
- `messages(session_id, created_at, id)`
- `ai_tasks(session_id, created_at)`
- `ai_tasks(status, created_at)`
- `ai_tasks(worker_id, created_at)`
- `story_states(story_id, updated_at)`
- `stories(status, updated_at)`
- `stories(author_id, updated_at)`
- `lore_entries(story_id)`
- `lore_entries(category)`
- `lore_entries(enabled)`

---

## 9. 第一版暂不做

暂不做：

- 多人协作权限。
- 复杂故事版本管理。
- 长期记忆向量库。
- Qdrant / Chroma 语义检索。
- 消息全文搜索。
- AI 任务自动重试队列。
- 多 Worker 调度历史统计。
- 用户付费和用量计费。
- 复杂多 Agent 协作状态。
