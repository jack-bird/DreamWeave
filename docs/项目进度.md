# 织梦纪 DreamWeave 项目进度

更新日期：2026-05-30

## 1. 当前阶段

当前项目已经跑通第一版核心线上闭环，并完成第一版 PostgreSQL 持久化代码接入。互动小说 Agent 引擎已完成第一版代码接入，当前本地仓库已补充 StoryState、AgentState、Lore 文件读取、最小 Agent 工作流、自动修订、Server 侧 `story_states` 读写接口，以及多故事 / 多会话管理能力。

新的产品方向已从“单用户 AI 小说工具”升级为“AI 互动小说平台”：保留现有 Node.js Server + Python Worker + Ollama 架构，新增作者端、玩家端、作品大厅、世界观编辑器和 RAG 检索系统。作者负责创建作品和维护世界观；玩家从大厅进入作品，创建独立存档并持续推进自己的 StoryState。

第一版已经围绕以下核心链路展开：

```text
用户输入 → 服务器转发 → 本地 Worker → Ollama 生成 → 返回剧情
```

目前已经完成本地 `Ollama + Python Worker` 基础能力、线上 `apps/server` 部署、第一版 H5 页面、数据库设计和初始化 SQL，以及 `apps/server` 的第一版 PostgreSQL 持久化代码。

后续升级方向已经明确：不推倒现有 Node.js Server + Python Worker 架构，而是把互动小说 Agent 引擎接入到 Python Worker 内部。Server 继续负责 API、WebSocket、PostgreSQL、会话、故事和状态持久化；Worker 负责 Agent 工作流、StoryState 解释、Lore 检索、Prompt 编排和 Ollama 调用。

目标升级链路：

```text
用户输入
        ↓
Server 读取 messages + StoryState
        ↓
Server 下发 AgentTask
        ↓
Worker 执行 Agent 工作流
        ↓
Worker 调用 Ollama 生成正文和 state_update
        ↓
Server 保存 messages / ai_tasks / story_states
        ↓
H5 展示剧情
```

平台化目标链路：

```text
作者创建 Work / World / Lore
        ↓
发布作品到玩家大厅
        ↓
玩家进入作品并创建 PlaySession
        ↓
Server 初始化独立 StoryState
        ↓
Worker 基于 StoryState + Lore/RAG 生成剧情
        ↓
Server 保存消息、任务和状态
        ↓
玩家继续互动
```

公网最小闭环已经跑通：

```text
公网 H5 页面输入
        ↓
线上 Server HTTP 接收请求
        ↓
线上 Server 通过 WebSocket 下发 AI 任务
        ↓
本地 Worker 调用 Ollama
        ↓
Worker 返回 AI 结果
        ↓
线上 Server 返回给 H5 显示
```

## 6. 当前关键决策

### 6.1 模型选择

模型不应写死在 Worker 中。

当前规则：

```text
任务 model 字段优先
        ↓
Worker 默认模型兜底
        ↓
模型不存在则返回 MODEL_NOT_FOUND
```

这意味着后续可以按用户、故事、会话或前端选择动态切换模型。

### 6.2 第一版任务队列

第一版可以先用内存 pending task 跑通闭环。

Redis / BullMQ 建议在核心链路跑通后接入。

### 6.3 第一版数据库

数据库不必阻塞 WebSocket 联调。

建议先跑通：

```text
HTTP 请求 → WebSocket Worker → Ollama → HTTP 返回
```

再补：

```text
保存用户输入和 AI 回复
```

### 6.4 Prompt 策略

第一版 Prompt 先追求稳定输出，不追求复杂多 Agent。

当前重点：

- 输出干净。
- 不替用户做决定。
- 保持剧情推进。
- 方便前端直接展示。

### 6.5 Agent 接入策略

Agent 引擎不替换当前线上 Server。

当前决策：

```text
Node.js Server 继续负责 API / WebSocket / PostgreSQL / 会话和状态持久化
        ↓
Python Worker 升级为 Agent 执行器
        ↓
Ollama 继续负责本地模型推理
```

第一版 Agent MVP 先做：

- StoryState。
- AgentState。
- Worker 返回 `state_update`。
- Lore 文件读取。
- 简单质量检查。

LangGraph、Qdrant / Chroma 和长期记忆向量化后置。

### 6.6 StoryState 保存边界

长期保存的 StoryState 只保存故事状态。

不建议长期保存：

- `user_input`。
- `recent_history`。
- `retrieved_lore`。
- `final_response`。

这些字段应只存在于本轮 AgentState，或分别由 `messages`、Lore 检索和 `ai_tasks` 承担。

### 6.7 平台化升级边界

下一阶段不推翻现有 `stories` / `sessions` / `story_states` 闭环，而是在其上抽象作者作品和玩家存档。

当前建议：

- `Work` 表示作者发布的公共作品模板。
- `PlaySession` 表示玩家从某个作品创建出来的独立存档。
- `StoryState` 继续按 session 隔离，避免不同玩家进度互相污染。
- `LoreEntry` 优先保存到 PostgreSQL，再逐步同步到 Chroma。
- 第一版 RAG 必须按 `work_id` 或 `world_id` 过滤，并只召回 `enabled = true` 的条目。
- 第一版先做 `top_k = 5` 的简单召回，不做复杂多 Agent、NPC 自主行为和自动剧情树。

## 8. 当前风险和挑战

### 8.1 模型输出仍需持续调优

Prompt 已经明显收紧，但不同模型仍可能出现风格、长度、动作边界不稳定的问题。

后续需要在真实交互中持续调整 Prompt、生成参数和后处理规则。

### 8.2 任务可靠性尚未完成

当前还没有持久任务队列。

如果服务器或 Worker 中途断开，任务恢复、重试和状态追踪还需要后续实现。

### 8.3 多用户系统复杂度

当前第一版主要是单用户体验，后续扩展到多用户时需要考虑：

- 用户身份认证和授权
- 数据隔离和隐私保护
- 并发访问和性能优化
- 成本控制和资源管理

建议在完成核心功能后再逐步引入多用户支持。

### 8.4 Agent 化复杂度

Agent 引擎会引入新的状态同步和质量控制问题：

- StoryState 可能被错误更新。
- 多轮剧情摘要可能遗漏关键信息。
- Lore 检索可能引入无关设定。
- 多节点 LLM 流程会增加延迟和失败点。

建议先做单会话 Agent MVP，确认 `content + state_update` 闭环稳定后，再引入 LangGraph 和向量检索。

### 8.5 平台化数据模型迁移风险

现有系统已经有 `stories` / `sessions`，新规划引入 `works` / `worlds` / `play_sessions`。后续实现前需要先明确兼容策略，避免同一概念在数据库中重复表达。

建议优先选择一种渐进方案：

- 短期：复用现有 `stories` / `sessions` 验证玩家大厅和开始游玩体验。
- 中期：新增 `works` / `lore_entries`，把作者作品和玩家存档边界拆清楚。
- 长期：根据真实使用情况决定是否保留 `stories` 作为内部兼容层，或迁移到 `works` / `play_sessions`。

### 8.6 RAG 召回质量风险

RAG 会带来新的质量问题：召回无关 Lore、漏召关键设定、跨作品污染、embedding 成本和向量库同步失败。

第一版必须做到：

- 按 `world_id` 或 `work_id` 强过滤。
- 只召回 `enabled = true` 的 Lore。
- 保留 PostgreSQL 作为权威数据源。
- RAG 调试页能看到实际召回结果。
- Chroma 写入失败时不影响基础剧情生成闭环。
